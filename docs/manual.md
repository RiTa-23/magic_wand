# Magic Wand開発マニュアル

このドキュメントは「魔法の杖で家電を操作するアプリ（Magic Wand）」の開発メンバー向けのマニュアルです。
本プロジェクトは、赤外線カメラで杖の動き（反射光）を検知し、さらに音声認識で呪文を聞き取ることで、「杖を振る＋呪文を唱える」という魔法体験をスマートプラグ経由で家電操作に結びつけるシステムを構築します。

---

## 👥 チーム体制と役割分担

本プロジェクトは大きく分けて4つの技術領域が絡むため、**役割ごとの分業（モジュール化）** を意識して開発を進めます。

| 役割（Role）               | 担当領域             | 主なタスク                                                                                |
| -------------------------- | -------------------- | ----------------------------------------------------------------------------------------- |
| **🪄 Device (赤外線検知)** | 杖の動き検知システム | 赤外線カメラモジュールの制御、画像処理、ジェスチャー認識ロジックの実装                    |
| **🎙 Voice (音声認識)**    | 呪文認識システム     | Web Speech APIによるリアルタイム音声認識、呪文マッチングロジックの実装                    |
| **🔌 IoT (プラグ制御)**    | 家電操作システム     | スマートプラグのAPI（SwitchBot等）の呼び出し、ON/OFF制御ロジック                          |
| **💻 UI (フロントエンド)** | 画面・全体統合       | Next.jsでの画面実装、検知システムと家電操作APIを連携するBFF(Backends For Frontends)の実装 |

---

## 📁 ディレクトリ構造（App Router + モジュール駆動）

各担当が **「自分の担当ディレクトリだけ見れば開発・テストできる」** 状態を作るため、`src/features/` ディレクトリ配下にドメイン（役割）ごとの切り分けを行います。

```
src/
├── app/                  # UI担当：Next.jsのルーティング、本番用の画面・API
│   └── test/             # 🧪 【各担当のサンドボックス（テスト画面）】
│       ├── device/       # 👉 赤外線担当がカメラの映像や検知結果をテストする画面
│       ├── voice/        # 👉 音声認識担当がマイク入力や呪文認識をテストする画面
│       └── iot/          # 👉 IoT担当がボタン一つでプラグのON/OFFをテストする画面
│
├── components/           # UI担当：共通コンポーネント（ボタン、レイアウトなど）
├── features/             # ✨ 担当ごとの機能モジュール ✨
│   ├── device/           # 👉 【赤外線検知 担当領域】
│   │   ├── api/          # カメラ連携の処理や外部システム通信
│   │   ├── lib/          # 検知ロジック・ジェスチャーの判定アルゴリズム
│   │   └── types/        # ジェスチャーの型定義（Swipe, Circle など）
│   │
│   ├── voice/            # 👉 【音声認識 担当領域】
│   │   ├── api/          # Web Speech APIのラッパー・React Hook
│   │   ├── lib/          # 呪文マッチングロジック・呪文辞書
│   │   └── types/        # 音声認識の型定義（SpellResult, SpellDictionary など）
│   │
│   ├── iot/              # 👉 【スマートプラグ操作 担当領域】
│   │   ├── api/          # SwitchBot/Nature Remo等 APIクライアント
│   │   ├── lib/          # プラグのON/OFF制御ロジック
│   │   └── types/        # デバイスの型定義
│   │
│   └── game/             # 👉 (必要に応じて) 全体のゲーム・体験ロジック
│
├── tests/                # テストコード
│   ├── features/device/  # 赤外線検知の単体テスト
│   ├── features/voice/   # 呪文マッチングロジックの単体テスト
│   └── features/iot/     # プラグ制御APIのモックテスト
│
└── utils/                # プロジェクト全体で使う共通関数（日付処理など）
```

### なぜこの構造なのか？

- 各担当者が `src/features/自分の担当/` の中で完結してロジックを書けるため、**コードの競合（コンフリクト）が起きにくい**です。
- UI担当は `features/iot/` や `features/device/` の中身の複雑なロジックを知らなくても、公開されている関数を呼び出すだけで本番画面と連携できます。
- 赤外線担当・音声認識担当・IoT担当は、UI担当の完成を待つことなく `src/app/test/自分の担当/` に **自分専用のテスト画面（ダサくてもボタンとテキストだけでOKな画面）** を作り、実際の映像・音声認識・デバイス操作をブラウザ上で検証できます。

## 🛠 開発の流れ・ルール

### 1. インターフェース（型）を先に決める（これは一旦無視でOK）

並行開発を成功させるため、**「各機能がどういうデータを受け取り、何を返すか」** を最初に話し合って `types/` に定義します。

- _例：Device担当からUI担当には `{"gesture": "circle", "confidence": 0.95}` というJSONを返す、など_

### 2. Sandbox（テスト画面）での仮実装と検証

本物の赤外線カメラやスマートプラグの制御ロジックを書く際、Device担当やIoT担当は `src/app/test/` の下に自分専用のページを作り、ブラウザから簡単に呼び出してテストできるようにします。
UI担当はそれと並行して、本番用のリッチな画面のモックを作り進めます。

### 3. テスト駆動（Test）

特にIoT操作やジェスチャー判定は画面がなくても成り立つ純粋なロジックです。
`vitest` を用いて、`devbox run test` で自分のロジックが正しく動くか単体テストを書きながら進めましょう。

## 🚀 日々の開発手順

1.  **環境立ち上げ**

    ```bash
    devbox shell
    ```

2.  **ローカルサーバー起動**

    ```bash
    devbox run dev
    ```

3.  **コミット前のチェック**（必ず通すこと！）
    これで Lint, Format, Test がすべてパスするか確認します。
    `bash
    devbox run check
    `
